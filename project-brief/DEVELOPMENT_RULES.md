# DEVELOPMENT RULES
## עולם דיגיטלי כלכלי קהילתי

כללי העבודה הקבועים של הפרויקט. חלים על כל מפתח, כל AI, כל sprint.

---

## כלל 1 — פיצ׳ר גמור לפני פיצ׳ר חדש

**לא מתחילים פיצ׳ר חדש לפני שהקודם עובד עד הסוף.**

"עד הסוף" מוגדר כ:
- לוגיקה עובדת
- UI ברור למשתמש
- כפתורי פתיחה וסגירה
- פידבק ויזואלי לכל פעולה
- דרך חזרה ממכל מסך
- בדיקה ידנית שעברה

---

## כלל 2 — כל פיצ׳ר חייב UI מלא

כל פיצ׳ר שנבנה חייב לכלול:

- [ ] כפתור פתיחה ברור
- [ ] כפתור סגירה / חזרה
- [ ] טקסטים בעברית ברורים
- [ ] פידבק ויזואלי לכל פעולה (הצלחה / שגיאה)
- [ ] מצב loading אם יש API call
- [ ] מצב ריק (אם אין נתונים — להסביר למה)

---

## כלל 3 — Zoom In = Zoom Out

**אם יש Zoom In חייב להיות גם Zoom Out.**

- כל מצלמה שנכנסת צריכה דרך לצאת
- כל מסך שנפתח צריך X / חזרה / ESC
- כל אנימציה כיוון אחד → צריכה reverse
- Zoom In לאזור → Back to Globe חייב לעבוד

---

## כלל 4 — כניסה = יציאה

**אם יש כניסה למסך / רכב / מבנה — חייבת להיות דרך חזרה.**

- כניסה לרכב → F לכניסה, F ליציאה
- פתיחת פאנל → X לסגירה
- כניסה לאזור חדש → Back / מפה לחזרה
- כניסה לתפריט → ESC לביטול

---

## כלל 5 — אין רובוטים חופשיים בעולם

**רובוטיקה תופיע רק כ: מכונות, תחנות עבודה, שירותים אוטומטיים.**

מה מותר:
- מכונת כרייה סטטית / חצי-מכנית
- מדפסת בטון — תחנת עבודה
- תחנת חקלאות אוטומטית עם אנימציה
- מסוע הובלה

מה אסור:
- רובוטים שמסתובבים חופשי בעולם ליד השחקן
- דמויות רובוטיות ללא מטרה ברורה
- רובוטים "למראית עין" בלי פונקציה

---

## כלל 6 — Asset Pipeline חובה

**לא בונים עולם ריאליסטי מקוד בלבד.**

- כל terrain חייב להשתמש ב-textures אמיתיות (Megascans / Poly Haven)
- כל מבנה חייב FBX asset ולא קוד גאומטרי
- כל דמות חייבת Skeletal Mesh + Animation
- Prototype בקוד — מותר. פינאל בקוד — אסור.

קובץ ה-Asset Library: `project-brief/ASSET_LIBRARY_PLAN.md`

---

## כלל 7 — Backend נשאר מרכזי

**ה-backend לא נמחק, לא מוחלף, לא מדולג.**

ה-backend אחראי על:
- משתמשים + Auth
- טוקנים
- משאבים + inventory
- שוק + מסחר
- מינוי רובוטים / שירותים
- admin panel

כל client (UE5 / web) מדבר עם אותו backend.

---

## כלל 8 — הפרדה ברורה בין שכבות

| שכבה | תפקיד | מיקום |
|------|--------|--------|
| UE5 Client | משחק, ויזואלים, שליטה | `game-client-unreal/` |
| Web Prototype | דמו, דשבורד, בדיקות | `web-prototype/` |
| Backend | API, DB, Logic עסקית | `backend/` |
| Project Brief | תכנון, docs, rules | `project-brief/` |

**אסור לערבב בין השכבות:**
- לא כותבים game logic בbackend
- לא שמים business logic ב-client
- לא עושים DB calls ישירות מה-client

---

## כלל 9 — ניהול Sprint

כל sprint:
1. הגדרת מטרה ברורה בהתחלה
2. רשימת DoD (Definition of Done) לפני שמתחילים
3. בדיקה ידנית בסוף
4. תיעוד מה הושלם / מה נשאר
5. לא מתחילים sprint הבא לפני שה-DoD עבר

---

## כלל 10 — תקשורת ברורה עם AI / Claude

כשמבקשים עזרה מ-AI:
- להגדיר בדיוק: מה הפיצ׳ר, מה הקובץ, מה הבעיה
- לציין: "עד הסוף" — ולא רק לוגיקה
- לציין: "כולל UI" — אם רוצים UI
- לא לצפות לפיצ׳ר גמור אם לא הוגדרו DoD ברורים
- לוודא שה-AI לא מוסיף פיצ׳רים שלא בוקשו

---

## Definition of Done — כללי

פיצ׳ר נחשב גמור רק כש:

```
✅ הלוגיקה עובדת
✅ ה-UI קיים ומסביר את עצמו
✅ יש כפתור פתיחה ויש כפתור סגירה
✅ יש פידבק ויזואלי לכל פעולה
✅ אין crash ידוע
✅ נבדק ידנית
✅ מתועד ב-sprint notes
```
